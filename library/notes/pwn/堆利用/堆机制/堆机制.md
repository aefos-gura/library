## main_arena中的bin:
**顺序:tcache->fast->small->unsorted->large->binmap(后三个会发生切割)**
### fastbins
FIlO.头插
非循环单链表 (fd).
范围:**0x20** to **0x80**.
fast bin不会对free chunk进行合并操作.
只有tcachebins满或未开启时进入 .
![[notes/pwn/堆利用/堆机制/图/QQ20251223-083340.png]]

### smallbins
FIFO(fd进,bk出)
循环双链表(fd,bk)
范围:**~0x400**
最多62个
只从unsortedbins中整理时进入   
![[notes/pwn/堆利用/堆机制/图/QQ20251223-091143.png]]
### largebins
Best Fit
两个循环双链表(fd,bk,nextsize)
nextsize维护的主链表,按降size排序(从大到小,头插),每个size的第一个chunk有nextsize指针,同size的chunk放在fd指针方向的下一个位置(已有同size的chunk时,新chunk不写nextsize指针,且插在第一个chunk后面--脖插)
malloc时也优先将没有nestsize的chunk分配出去
范围:**0x400~** 
不限制最大数量
只从unsortedbins中整理时进入
![[notes/pwn/堆利用/堆机制/图/Pasted image 20251223091644.png]]
### unsortedbins
Best Fit,沿bk方向检索,头插.
范围:**0x90~**
循环双链表.
不限制最大数量
沿bk方向检索,头插.

### arena
管理上述bins的struct,在libc中.
![[notes/pwn/堆利用/堆机制/图/arena结构.png]]
该struct的前三个区域(标志位)及功能如下:
	+`mutex`:标志是否上锁,确保在复杂的堆内存操作中，数据结构不会因为多线程并发而崩溃.(锁的具体机制不过多解释)
	+`flags`:表示是否该arena中的chunk在连续的堆内存中.
	+`have_fastchunks`:为boolean,表示该arena使用过fastbins.
之后是fastbins的链表头(0x20~0xb0)
即  `mfastbinptr fastbinsY[NFASTBINS]`数组

`last_remainder`:chunk_ptr类型struct指针,指向上一个被切后剩余的chunk

之后是unsortedbins,smallbins和largebins的链表头和尾
这里共同由一个`mchunkptr bins[NBINS * 2 - 2]`数组表示

`binmap[]`:bitvector(位向量,指向依托boolean数组的数组指针),标志哪个largebins/smallbins中是有chunk的,用以在malloc时寻找size更大且有chunk的bins(在unsortedbin/largebin_search失败之后)

`next`:管理一个属于该arena的arena组成的单向循环链表
`next_free`:管理一个属于该arena的被free的arena组成的单向非循环链表(没有threads链接)
`Attached_threads`:正使用该arena的threads数
`System_mem`:该arena向系统mmap的总可写内存
`Max_system_mem`:该arena曾有过的最大可写总内存大小

## tcache
范围:0x10~0x410. 
FILO
共64个按大小分类的单链表.
每个单链表中最多有七个被free的chunk.
被free的chunk
最优先放进这个bins(包括在unsortedbins中被调整,重分配的).
![[notes/pwn/堆利用/堆机制/图/QQ20251226-161019.png]]
```
typedef struct tcache_perthread_struct
{
  uint16_t counts[TCACHE_MAX_BINS];
  tcache_entry *entries[TCACHE_MAX_BINS];
} tcache_perthread_struct;

/* We overlay this structure on the user-data portion of a chunk when
   the chunk is stored in the per-thread cache.  */
typedef struct tcache_entry
{
  struct tcache_entry *next;
  /* This field exists to detect double frees.  */
  uintptr_t key;
} tcache_entry;
```

## 堆中的一些操作

### unlink/link
操作链表,将某个victim(要处理的chunk)进行解链,上链的操作

### Exhausting(疲倦)
不会切割比需要size只大0x10的chunk,因为0x10不够最小大小(0x20),只能放chunk头,不够放userdata,所以会直接将该chunk进行分配.

### Consolidation(合并)
在一个chunk被free时进行,查看其前后是否为被free的chunk,是就进行合并.
fast bin不会对free chunk进行合并操作.

### malloc_hook
在glibc>=2.34已被移除qwq,介绍如下
	1. `__malloc_hook`/ `__free_hook` / `__realloc_hook`:在glibc中,默认为0.但被设置后,会在函数(malloc,free,realloc)被调用时,调用hook(钩子函数)处的内容,比如one_gadget.
	2. `__malloc_initialize_hook`:会在第一次malloc(对malloc进行初始化.initialize.时)调用.
	3. `__memalign_hook`:调用`aligned_alloc()`, `memalign()`, `posix_memalign()`, `valloc()`函数时,会调用这个内存对齐hook.
	4. `_dl_open_hook`:在abort message(终止信息)出现时,会调用`init()`中的`__backtrace()`,该函数会调用`__libc_dlopen()`和`__libc_dlsym()`,而`_dl_open_hook`指向了一个vtable(虚表函数),以上两函数通过该vtable触发(`_dl_open_hook`->`dlopen_mode`和`_dl_open_hook`->`dlsym`).
	5. `__after_morecore_hook`:在`sbrk()`函数申请完内存后调用该hook.
